## Context

当前仓库处于《猜历史人物》v3 的规划阶段。产品目标是做一个移动端优先的简体中文多人实时问答游戏：所有玩家参与同一个当前谜题，通过提问缩小范围，AI 只能返回 `是`、`不是`、`不确定`、`无关`、`猜对了` 五种判定。

产品结构包含两个页面和三个弹窗：入场弹窗、主页大厅、问答页、历史问答弹窗、猜对结果弹窗。游戏状态和问答历史必须持久化，刷新页面后不能丢失，并且所有在线玩家需要看到一致的状态。

## Goals / Non-Goals

**Goals:**

- 建立一个全局共享的当前回合，让所有玩家看到同一个当前人物和同一份问答时间线。
- 持久化玩家展示身份、当前回合、问答记录、已猜出人物历史。
- 串行处理提问，确保一个问题只有在 AI 判定完成后，才作为完整条目进入时间线。
- 对 AI 输出做规范化，只允许五个固定判定值进入数据库和 UI。
- 实现主页、问答页，以及入场、历史问答、猜对结果三个弹窗的状态流转。
- 每次新题由 AI 生成中国古代人物，并把已猜出人物作为上下文，尽量避免重复。

**Non-Goals:**

- 不做账号、密码、OAuth 或跨设备身份恢复。
- 不做房间、私局、匹配、排行榜或个人积分系统。
- 不使用预设人物题库。
- 不展示自由聊天内容，也不展示 AI 的解释性文本。
- 不专门设计复杂桌面端布局，只做移动端优先的响应式适配。

## Decisions

### 使用本地匿名身份

玩家首次进入时选择 emoji 头像和昵称，确认后保存在本地。之后每次提交问题时带上这份展示身份，用于时间线里的提问者展示。

备选方案是使用数据库匿名 session，但在没有登录体系的前提下，它会增加恢复和管理复杂度，收益不高。

### 以“回合 + 完整问答条目”为核心数据模型

每个谜题对应一个 game round，保存隐藏人物、回合状态、提问数量和时间信息。每条问答记录作为 round 的子记录，包含问题、提问者展示身份、序号、AI 判定和完成时间。

没有采用“先展示问题、稍后补 AI 答案”的方式，因为这会违反产品要求：每条问答必须作为整体条目展示，不能出现碎片化列表。

### 通过服务端动作处理提问

提问提交到后端 action 或 API，由服务端校验身份和输入、读取当前回合、调用 AI、规范化判定、分配序号、保存完整问答，并在 `猜对了` 时标记回合完成。

不让客户端直接调用 AI，因为那会暴露隐藏人物和提示词，也难以统一控制 AI 输出格式和并发顺序。

### 直接揭晓走独立分支

如果玩家明确表达“想直接知道答案”“不想猜了”之类意图，系统不再把它当作普通猜题问题，而是走揭晓分支：记录当前答案、标记当前回合已揭晓，并立即创建新的当前回合。

这样做可以避免把“求答案”误塞进普通判定范围，同时保证下一轮和刚刚揭晓的答案不同。

### UI 从持久化状态同步

客户端进入页面时先从数据库加载当前状态，再订阅当前回合、问答记录和已猜出历史的变化。时间线只渲染已经持久化的完整问答记录。

只依赖本地 WebSocket 事件的方案刷新恢复能力较差，因此不作为状态来源。

### 猜对后由结果弹窗承接下一题

当回合被猜对时，所有客户端弹出猜对结果弹窗。点击 `再猜一个` 时再创建或加入下一个未完成回合；点击 `返回主页` 时只关闭弹窗并回到主页。

没有选择“猜对后立刻自动生成下一题”，因为那会让玩家还没看清答案时，主页和当前题就已经切到下一轮，体验容易混乱。

### 维持轻量古风样式系统

整体使用浅黄色宣纸底色、墨色文字、低饱和朱砂点缀和移动端优先布局。主页中央 `?` 是主要视觉，但呈现上更像“未揭开的题签”或“卷轴封面”；按钮可采用印章感，弹窗可采用卷轴展开感，问答时间线可借鉴竹简或史书记载的纵向节奏。

古风只作为交互和视觉氛围，不引入复杂装饰。界面仍要少卡片、留白充足、点击区域明确，避免因为纹理、边框或拟物效果影响可读性。

## Risks / Trade-offs

- AI 返回了不合规文本 -> 服务端做输出规范化，无法识别时重试或回退为 `不确定`。
- AI 把朝代、性别等相关问题误判为 `无关` -> 提示词加入更明确的相关范围说明，并在后端做轻量兜底。
- AI 生成了重复人物 -> 生成提示词包含所有已猜出人物，并在入库前检查重复。
- 多个玩家同时提问 -> 服务端统一分配序号，并在更新当前回合时使用事务或等价锁机制。
- 玩家想直接看答案 -> 揭晓分支直接结束当前回合并开新回合，且新回合生成时继续避开刚刚揭晓的人物。
- AI 响应较慢 -> 提交者输入区显示 `AI 判定中...`，并阻止重复发送；时间线不提前插入半成品。
- 多个玩家同时点击 `再猜一个` -> 创建下一回合时先检查是否已有未完成回合，保证幂等。
- 实时订阅中断 -> 页面刷新或重连时从数据库重新拉取完整状态。

## Migration Plan

1. 建立持久化结构：游戏回合、问答记录、已猜出人物历史。
2. 实现后端动作：加载初始状态、提交问题、创建下一回合、读取历史问答。
3. 实现主页、问答页、三个弹窗和实时订阅。
4. 接入 AI：生成中国古代人物、判定玩家问题、规范化输出。
5. 验证刷新恢复、双客户端同步、猜对弹窗广播、下一题并发保护。

如果需要回滚，可以移除 v3 页面和后端动作，同时保留数据库中的历史数据不使用。若数据库只是开发环境，也可以按迁移策略清理相关表。
